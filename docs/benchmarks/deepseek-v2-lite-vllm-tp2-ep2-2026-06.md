# DeepSeek-V2-Lite EP2: OpenInfer and vLLM TP2/EP2 Snapshot

**Created**: 2026-06

**TL;DR**: 2026-06-28 DeepSeek-V2-Lite EP2 run on 2x RTX 5090. HF, OpenInfer host-staged, and OpenInfer NCCL matched exactly on the committed case set; OpenInfer host-staged/NCCL completed direct diagnostics, HTTP pressure, and trace rows. Stock vLLM TP2 and TP2+EP2 failed at server startup on the FlashInfer SM120/CUDA 12.8 path. A separate FlashInfer #3633-equivalent validation completed the vLLM HTTP rows under the same client/workload contract. Treat this as a retained benchmark snapshot and setup-gap record, not a vLLM parity or production serving claim.

Source benchmark for [#279](https://github.com/openinfer-project/openinfer/issues/279) and the DeepSeek-V2-Lite status ledger.

## References

- [vLLM `bench serve` CLI](https://docs.vllm.ai/en/latest/cli/bench/serve/) defines `--max-concurrency` as concurrent request pressure and says `--request-rate inf` sends all requests at time 0.
- [vLLM Expert Parallel Deployment](https://docs.vllm.ai/en/latest/serving/expert_parallel_deployment/) marks EP experimental and enables it with `--enable-expert-parallel`.
- [vLLM GPU installation docs](https://docs.vllm.ai/en/latest/getting_started/installation/gpu/) note default CUDA 12.9 vLLM binaries and specific CUDA 12.8 / 13.0 wheel options for NVIDIA users.
- [FlashInfer README](https://github.com/flashinfer-ai/flashinfer) lists CUDA 12.8 support and JIT/cache install paths.
- [FlashInfer PR #3633](https://github.com/flashinfer-ai/flashinfer/pull/3633), [FlashInfer issue #3675](https://github.com/flashinfer-ai/flashinfer/issues/3675), and [vLLM issue #42393](https://github.com/vllm-project/vllm/issues/42393) match the SM120 / CUDA 12.8 FlashInfer JIT failure seen in the stock vLLM rows.

## Reproduction Commands

Print the matrix plan:

```bash
python3 scripts/bench_dsv2lite_vllm_matrix.py --plan-only
```

Run the full retained matrix:

```bash
python3 scripts/bench_dsv2lite_vllm_matrix.py \
  --model-path models/DeepSeek-V2-Lite \
  --out-dir target/benchmarks/deepseek-v2-lite-vllm-tp2-ep2 \
  --hf-python <python-with-transformers-4.40.x> \
  --openinfer-trace-pass \
  --keep-going
```

Use an explicit `--hf-python` when the HF oracle needs a different Transformers environment from the vLLM server environment.

Summarize an existing artifact directory without rerunning benchmarks:

```bash
python3 scripts/bench_dsv2lite_vllm_matrix.py \
  --summarize-only target/benchmarks/deepseek-v2-lite-vllm-tp2-ep2/<timestamp> \
  --hf-python <python-with-transformers-4.40.x>
```

The script writes `summary.json` and keeps raw child artifacts. Summaries are derived from JSON artifacts and server logs, so failed setup rows remain visible even when a server never produced per-cell JSON.

## Claim Boundaries

- `vllm bench serve --max-concurrency` is HTTP concurrency pressure.
- OpenInfer direct same-prompt batch `1/4/8` is diagnostic model-path evidence.
- OpenInfer trace rows can prove `decode_batch_size_max` for OpenInfer only.
- OpenInfer `host-staged` and `NCCL` are backend/oracle labels. vLLM `TP2` and `TP2+EP2` are baseline labels. The dimensions are not symmetric.
- Clean failed setup rows are part of the retained evidence. They must not be omitted from a comparison table.
- The supplemental vLLM validation run proves the same HTTP client/workload contract can run after the FlashInfer SM120 / CUDA 12.8 JIT fix. It is not a stock package-stack claim.
- This snapshot does not claim vLLM parity, sparse-dispatch readiness, multi-node EP, full CUDA Graph serving coverage, NCCL HTTP serving readiness, or production DeepSeek-V2-Lite serving readiness.
